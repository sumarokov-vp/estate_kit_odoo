# Комиссия

## Обзор

Система рассчитывает ожидаемую комиссию на всех этапах воронки продаж — от квалификации лида до закрытия сделки. По мере продвижения лида по воронке точность расчёта увеличивается: от приблизительной оценки на основе фильтров клиента до точной суммы при закрытии сделки.

```
Лид создан          Объект выбран          Сделка закрыта
    │                    │                       │
    ▼                    ▼                       ▼
 Оценка               Точная                  Факт
 (фильтры +           (listing_price ×        (listing_price ×
  AI-прогноз)          % комиссии)             commission_percent)
```

---

## Защита от занижения цены

### Проблема

При осмотре объекта покупатель и продавец могут договориться о цене ниже публикационной. Это создаёт вектор обхода комиссии: стороны искусственно занижают официальную цену сделки, чтобы уменьшить сумму комиссии.

### Решение: listing_price как база для комиссии

Комиссия всегда считается от **цены публикации объекта на MLS** (`listing_price`), а не от фактической цены сделки (`deal_price`).

- `listing_price` — цена на момент публикации в MLS. Фиксируется при подаче объекта и не может быть изменена без повторной модерации.
- `deal_price` — фактическая цена, по которой закрылась сделка. Может отличаться от `listing_price` в результате торга.

```
listing_price = 50 000 000       ← база для комиссии (неизменяема)
deal_price    = 47 000 000       ← фактическая цена после торга
commission    = 50 000 000 × 3%  = 1 500 000  ✓ (не 47M × 3%)
```

### Допустимый коридор снижения

Снижение цены сделки относительно `listing_price` контролируется параметром `max_discount_percent`:

- **Ключ:** `estate_kit.max_discount_percent`
- **По умолчанию:** 10%

Если `deal_price` ниже `listing_price` на величину, превышающую допустимый коридор — система предупреждает менеджера. Это не блокирует сделку, но сигнализирует о возможном занижении.

## Источники ставки комиссии

Ставка комиссии для лида определяется по приоритету:

| Приоритет | Источник | Когда применяется |
|-----------|----------|-------------------|
| 1 | MLS-комиссия объекта | Объект привязан к лиду и получен из MLS |
| 2 | Минимальная комиссия агентства | Настройка модуля (фиксированный %) |

### MLS-комиссия

MLS устанавливает минимальную комиссию на объект (`min_commission_percent`). Комиссия не может быть ниже этого порога — от неё исчисляется вознаграждение листинг-агенту. Ставка привязана к объекту и имеет приоритет над настройкой агентства.

### Минимальная комиссия агентства

Настройка в `ir.config_parameter`:

- **Ключ:** `estate_kit.min_commission_percent`
- **Тип:** Float (процент)
- **UI:** Настройки → Estate Kit → блок "Комиссия"

Применяется, если у объекта нет MLS-комиссии или объект собственный.

## Use Case: получение комиссии для лида

Единая точка входа для получения применимой ставки комиссии. Сейчас — простая логика (настройка или MLS), в будущем — расширяемая (индивидуальные ставки, тарифные планы, типы сделок).

### Логика

```
1. Если у лида есть привязанный объект:
   a. Если объект из MLS и у него есть MLS-комиссия → вернуть MLS-комиссию
   b. Иначе → вернуть минимальную комиссию агентства из настроек
2. Если объекта нет → вернуть минимальную комиссию агентства из настроек
```

### Архитектура (DDD-сервис)

```
src/lead/services/commission/
├── __init__.py
├── factory.py              # Composition root
├── service.py              # Фасад: get_rate(lead) → float
├── protocols/
│   ├── __init__.py
│   └── i_rate_provider.py  # Protocol: get_rate(lead) → float
└── settings_rate_provider.py  # Реализация: читает ir.config_parameter
```

Фасад пока делегирует единственному `IRateProvider`. При появлении MLS-ставок, индивидуальных тарифов — добавляются новые реализации, фабрика собирает цепочку приоритетов.

## Расчёт ожидаемого дохода на лиде

Поле `expected_revenue` на `crm.lead` — **вычисляемое** (computed, not stored). Показывает прогноз комиссии в денежном выражении.

### Формула

```
expected_revenue = estimated_price × commission_rate / 100
```

Где:

- **commission_rate** — результат use case `get_rate(lead)`
- **estimated_price** — оценочная стоимость объекта (см. ниже)

### Определение estimated_price

По приоритету:

| Приоритет | Источник | Условие |
|-----------|----------|---------|
| 1 | `property_id.listing_price` | Объект привязан к лиду (цена публикации, не сделочная) |
| 2 | Среднее из фильтров клиента | `search_price_min` и/или `search_price_max` заданы |
| 3 | AI-оценка средней стоимости | Фильтры цены не заданы |

#### Среднее из фильтров

```python
if search_price_min and search_price_max:
    estimated_price = (search_price_min + search_price_max) / 2
elif search_price_max:
    estimated_price = search_price_max
elif search_price_min:
    estimated_price = search_price_min
```

#### AI-оценка средней стоимости

Когда у клиента нет фильтров по цене, но есть другие критерии поиска (тип, район, площадь), система может запросить у AI приблизительную рыночную стоимость.

!!! warning "AI-модуль должен быть заменяемым"
    Интеграция с AI (сейчас Anthropic Claude) оформлена как отдельный модуль с протоколом. При необходимости реализация заменяется на другого провайдера (OpenAI, локальная модель) без изменения бизнес-логики.

**Входные данные для AI:**

- `search_property_type` — тип недвижимости
- `search_city_id` — город
- `search_district_ids` — районы
- `search_rooms_min` / `search_rooms_max` — количество комнат
- `search_area_min` / `search_area_max` — площадь

**Архитектура AI-клиента:**

Текущий `AnthropicClient` в `src/shared/services/anthropic_client.py` — конкретная реализация. Для заменяемости нужен протокол:

```
src/shared/services/ai_client/
├── __init__.py
├── protocols/
│   ├── __init__.py
│   └── i_ai_client.py       # Protocol: complete(system, user) → str | None
├── anthropic_client.py       # Текущая реализация (Anthropic API)
└── factory.py                # Создаёт клиента по настройкам
```

Все сервисы, использующие AI (скоринг, оценка стоимости), зависят от `IAiClient`, а не от `AnthropicClient` напрямую.

## Хранение данных

### Принцип: вычисляемые поля не хранятся

`expected_revenue` — **computed, not stored**. Пересчитывается при изменении зависимостей:

```python
@api.depends('property_id.listing_price', 'search_price_min', 'search_price_max')
def _compute_expected_revenue(self):
    ...
```

### Поля на crm.lead

| Поле | Тип | Stored | Описание |
|------|-----|--------|----------|
| `expected_revenue` | Monetary | No (computed) | Ожидаемая комиссия |

### Поля на estate.property

| Поле | Тип | Описание |
|------|-----|----------|
| `listing_price` | Monetary | Цена публикации на MLS (фиксируется, не меняется без повторной модерации) |
| `min_commission_percent` | Float \| null | Минимальная комиссия MLS (%) |

### Новые настройки модуля

| ir.config_parameter | Тип | Описание |
|---------------------|-----|----------|
| `estate_kit.min_commission_percent` | Float | Минимальная комиссия агентства (%) |
| `estate_kit.max_discount_percent` | Float | Допустимый коридор снижения цены сделки от listing_price (%, по умолчанию 10) |

### Поля на estate.deal

| Поле | Тип | Описание |
|------|-----|----------|
| `deal_price` | Monetary | Фактическая цена сделки (может отличаться от listing_price) |
| `commission_percent` | Float | Фактический % комиссии |
| `total_commission` | Monetary (computed) | Комиссия от `listing_price × commission_percent` (не от deal_price) |
| `price_discount_warning` | Boolean (computed) | True, если снижение deal_price > max_discount_percent |

## Этапы реализации

### Этап 1: Базовая комиссия

- [ ] Настройки: `min_commission_percent`, `max_discount_percent` в settings
- [ ] Поле `listing_price` на `estate.property` (фиксируется при публикации)
- [ ] Use case `CommissionService.get_rate(lead)`
- [ ] Computed `expected_revenue` на `crm.lead` (на основе `listing_price` или фильтров)
- [ ] Computed `price_discount_warning` на `estate.deal`
- [ ] `total_commission` считается от `listing_price`, не от `deal_price`

### Этап 2: MLS-комиссия

- [ ] Поле `min_commission_percent` на объекте (из MLS API)
- [ ] `MlsRateProvider` — учёт MLS-ставки в цепочке приоритетов

### Этап 3: AI-оценка стоимости

- [ ] Протокол `IAiClient` — абстракция над AI-провайдером
- [ ] Рефакторинг `AnthropicClient` под протокол
- [ ] Сервис оценки средней стоимости по критериям поиска
- [ ] Интеграция в расчёт `expected_revenue` как fallback
